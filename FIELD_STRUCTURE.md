# ALIS Form Markup Field Structure Guide

## Overview

When you upload a PDF form, the system analyzes each form field and extracts/generates a set of properties. Understanding what each property means and where it comes from is critical to using the system effectively.

---

## Field Properties Table

| Property | Type | Source | Purpose | Example |
|----------|------|--------|---------|---------|
| **Field Name** | String | PDF form field name | The actual technical name of the form field in the PDF | `Text2`, `CheckBox1`, `residentSignature` |
| **Suggested Code** | String | ALIS code matching | The compliance/regulatory code matched to this field | `alis.resident.full_name`, `generic.date_assessment.1` |
| **Signer** | Enum | User configuration + prediction | Who signs/completes this field | `Resident`, `Staff`, `Manager`, `Unassigned` |
| **Anchor** | String | Generated from field metadata | A unique reference point to locate this field if the PDF changes | `text2.text.1`, `resident_sig.signature.1` |
| **Required** | Boolean | Code configuration or field attributes | Whether this field must be filled to submit | `true`, `false` |
| **Read-Only** | Boolean | Code configuration or field attributes | Whether the field is protected from editing | `true`, `false` |
| **Confidence** | Percentage | Fuzzy matching algorithm | How confident the system is in the suggested code match | `65%`, `92%`, `45%` |

---

## Data Flow & Origin

### 1. Field Name (PDF Form Field Name)

**Where it comes from:**
- Extracted directly from the PDF's form field dictionary using `pdf-lib`
- Examples: `Text2`, `topmostSubform[0].residentName`, `CheckBox1_Agree`

**What the app can do with it:**
✅ **Read**: Yes - fully extracted by field-detector.js  
⚠️ **Edit**: Currently NO (but could be enabled)  
📍 **Purpose**: Primary identifier; used as anchor and in documentation

**In the form-markup table:**
- Displays in the "Field Name" column
- Used to generate anchor references
- Helps users identify which PDF field they're configuring

---

### 2. Suggested Code (ALIS/Compliance Code)

**Where it comes from:**
- Generated by fuzzy matching the field's extracted label against ALIS master codes
- Process:
  1. Extract clean label from field name (e.g., "Text2" → "Text 2")
  2. Run fuzzy match against ~2000+ ALIS compliance codes
  3. Return best match with confidence score

**Example matching:**
- Field name: `residentFullName` → Cleaned: "Resident Full Name" → Match: `alis.resident.full_name` (95% confidence)
- Field name: `medicalNotes` → Cleaned: "Medical Notes" → Match: `generic.text_medical_notes.1` (78% confidence)

**What the app can do with it:**
✅ **Read**: Yes  
✅ **Edit**: Yes - users can override in the review table  
📍 **Purpose**: Links form fields to compliance requirements

**In the form-markup table:**
- Editable text input field
- Users can manually correct if auto-match is wrong
- Sent to property-applier when "Apply & Download" is clicked

---

### 3. Signer (Person/Role)

**Where it comes from:**
- **Initially**: Set to "Unassigned" by default
- **Configured by user**: In Upload page's "Configure Signers/Anchors" section
- **Future enhancement**: Could be predicted from nearby text (e.g., "Resident Signature" → predict "Resident")

**Available signers (user-configurable):**
- Defaults: `Resident`, `Staff`
- Custom: User can add any (e.g., `Manager`, `Physician`, `Witness`, `Administrator`)

**What the app can do with it:**
✅ **Read**: Yes  
✅ **Edit**: Yes - dropdown in review table  
⚠️ **Predict**: Not yet (future enhancement)  
📍 **Purpose**: Track who signs/completes each field for compliance workflows

**In the form-markup table:**
- Dropdown selector populated from configured signers
- Used during PDF application to route fields to correct signers

---

### 4. Anchor (Reference Point)

**Where it comes from:**
- Generated by `generateAnchor()` function in form-markup.js
- Format: `{fieldName}.{fieldType}.{pageNumber}`
- Example: `text2.text.1`, `resident_sig.signature.1`

**Purpose:**
- Creates a stable reference to find a field even if the PDF is modified
- Used by property-applier to locate and update fields
- Similar to a "bookmark" or "selector" in the PDF

**Generation logic:**
```javascript
// Field: "Text2" (text type, page 1)
// Generated anchor: "text2.text.1"

// Field: "residentSignature" (signature type, page 2)  
// Generated anchor: "resident_signature.signature.2"
```

**What the app can do with it:**
✅ **Read**: Yes  
✅ **Edit**: Yes - text field in review table  
📍 **Purpose**: Stable reference for property-applier to locate fields

**In the form-markup table:**
- Editable text input
- Users can customize the anchor if needed
- Should be unique per document

---

### 5. Required (Boolean)

**Where it comes from:**
- Determined by code configuration in master list
- Falls back to field attributes in the PDF
- Overridable by user

**Example:**
- Code `alis.resident.full_name` → Always required by default
- Code `generic.text_notes.1` → Required only if configured in ALIS definition

**What the app can do with it:**
✅ **Read**: Yes  
✅ **Edit**: Yes - checkbox in review table  
📍 **Purpose**: Indicates mandatory fields during form completion

**In the form-markup table:**
- Checkbox input
- Checked = field is required
- Unchecked = field is optional

---

### 6. Read-Only (Boolean)

**Where it comes from:**
- Determined by code configuration (e.g., auto-calculated fields)
- Falls back to field attributes in the PDF
- Overridable by user

**Example:**
- Code `generic.calculated_total.1` → Read-only (system calculates)
- Code `generic.text_notes.1` → Editable (user enters)

**What the app can do with it:**
✅ **Read**: Yes  
✅ **Edit**: Yes - checkbox in review table  
📍 **Purpose**: Prevents accidental modification of system-controlled fields

**In the form-markup table:**
- Checkbox input
- Checked = field is protected/read-only
- Unchecked = field is editable

---

### 7. Confidence (Percentage)

**Where it comes from:**
- Generated by the fuzzy matching algorithm in `code-matcher.js`
- Confidence thresholds:
  - **95%+**: Auto-approve (high confidence match)
  - **80-94%**: Likely correct (review recommended)
  - **70-79%**: Manual review suggested
  - **<70%**: Manual review required

**Example:**
- "Resident Full Name" → matches "alis.resident.full_name" → 95% confidence
- "Notes" → matches "generic.text_notes.1" → 65% confidence  
- "Xyz123Field" → no good match → 0% confidence

**What the app can do with it:**
✅ **Read**: Yes  
❌ **Edit**: No (auto-calculated, read-only)  
📍 **Purpose**: Indicates how much to trust the auto-suggested code

**In the form-markup table:**
- Visual progress bar + percentage text
- Used to guide user review priority
- Higher confidence = less likely to need manual correction

## Example Workflow

### Step 1: Upload PDF
```json
{
  "pdf": "base64-encoded-pdf",
  "company_name": "Bethesda Senior Living",
  "document_title": "Assisted Living Regulations Concerning Medications",
  "signers": ["Resident or Responsible Party", "Staff"]
}
```

### Step 2: System Analyzes
1. **field-detector**: Extracts "Text2", "CheckBox1", etc. from PDF
2. **label-extractor**: Cleans to "Text 2", "Check Box 1"
3. **property-suggester**: Matches to ALIS codes with confidence scores
4. **form-markup.js**: Stores in database with anchors

### Step 3: Database Storage
```javascript
{
  field_name: "Text2",           // From PDF
  field_type: "text",            // From PDF
  suggested_code: "generic.text_medication_name.1",  // From matcher
  signer: "unassigned",          // Default
  anchor_name: "text2.text.1",   // Generated
  required: false,               // From code config
  read_only: false,              // From code config
  confidence: 0.78,              // From matcher
  approval_status: "review_needed"  // From confidence
}
```

### Step 4: User Review & Edit
- User sees "Text2" in review table
- User can:
  - Change suggested code if wrong
  - Assign to a signer (Resident, Staff, etc.)
  - Modify anchor if needed
  - Check/uncheck Required and Read-Only
- Clicks "Apply & Download"

### Step 5: Property Applier
- Uses field_name to locate field in PDF
- Applies properties based on user edits
- Outputs modified PDF

## Troubleshooting

### "Field Name is empty or shows 'undefined'"
- **Cause**: PDF field doesn't have a proper name set
- **Solution**: The system should still detect the field and create an anchor; review and adjust as needed

### "Confidence is 0%"
- **Cause**: Field label doesn't match any known ALIS codes
- **Solution**: Manually enter the correct code or select from dropdown suggestions

### "Anchor shows 'field.undefined.1'"
- **Cause**: Field type wasn't properly detected
- **Solution**: Check the field in PDF properties; the system should have assigned a type

### "Signer dropdown is empty"
- **Cause**: No signers configured during upload
- **Solution**: Re-upload the PDF and configure signers in the "Configure Signers/Anchors" section
